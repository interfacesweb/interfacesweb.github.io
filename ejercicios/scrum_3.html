1 Introduction

We are going to create the visual design for a new site using the Bootstrap framework.

Teams will create a responsive web site using CSS classes, components and JavaScript provided by Bootstrap.

Each team will have a GIT repository for working and deploying their sites. The main branch of the repository will be gh-pages so that the site can be available at github.io.

2 Learning goals

  • Develop web interfaces according to given requirements.
  • Learn how to use development frameworks to create web sites.
  • Develop responsive web interfaces that can be displayed correctly in all kinds of devices and screens.
  • Learn how to work in teams using a version control system.

4 Product Backlog

Each team must create an online store. Teams can decide the type of store (sporting goods, clothes, technology,…) they are going to develop.

Teams will use GitHub to keep track of their code and GitHub pages to deploy it.

Only the front-end will be developed. There will be no server logic or database.

4.1 Reading: documentation

Every team member must read the following sections of the documentation available in http://getbootstrap.com:

  • Getting started.
  • Grid system.
  • Tables.
  • Images.
  • Forms.

In addition to this, you should check some of the code in these sections too:

  • Components.
  • Javascript.

Finally, have a look at some Bootstrap designs in http://builtwithbootstrap.com/ and http://expo.getbootstrap.com/.

Definition of Done
You have read and understood the documentation.

4.2 Video: Bootstrap

Every member of the team must view this video tutorial about Bootstrap.

Definition of Done

4.3 Visual appearance

  • The site must look homogeneous. It should be personalized (logos, colors, fonts, icons,…). You can use the standard Bootstrap theme or get other themes from Internet.
  • All pages must be responsive. They must be different for:
    • Mobile
    • Tablet
    • Other screens
Definition of Done
Each page is displayed correctly and meets the style specifications the team has decided. Each page is available in GitHub Pages. Each page has been tested for the three kinds of devices. All team members are satisfied with the design of the website.

4.4 Page sections

The site must be organized and it must have at least these sections:

  • Header
  • Footer
  • Navigation menus
Definition of Done
Each page has the three required sections.

4.5 Main page

It must show the name, logo and main info of the site. It should also display the most important info of the site (sales, special products, deals,…).

At least it must include a link to the list of categories page.

Definition of Done
The page is deployed in GitHub pages. The page meets the proposed requirements. All members are satisfied with the result. The page is displayed correctly on the three kinds of devices.

4.6 List of categories page

It must show all the categories of items of the store. At least there must be 4 categories of items. Each category must have:

  • Description.
  • Image.
  • Link to its category page.
Definition of Done
The page is deployed in GitHub pages. The page meets the proposed requirements. All members are satisfied with the result. The page is displayed correctly on the three kinds of devices.

4.7 Category pages

  • There must be at least 4 pages of categories (one for each category).
  • Each page must show at least 6 items.
  • Each item must show (at least):
    • Name.
    • Image.
    • Price.
    • Link to the item page (it it exists).
Definition of Done
The pages are deployed in GitHub pages. The pages meet the proposed requirements. All members are satisfied with the result. Each page is displayed correctly on the three kinds of devices.

4.8 Item pages

  • There must be at least 8 item pages (two of each category). You can choose 2 of the 6 items of each category.
  • Each page must show (at least):
    • Item name.
    • 2 item images (at least).
    • Price.
    • Link or button to buy.
    • Client opinions.
Definition of Done
The pages are deployed in GitHub pages. The pages meet the proposed requirements. All members are satisfied with the result. Each page is displayed correctly on the three kinds of devices.

4.9 Contact page

It must have a form to contact customer service.

Definition of Done

4.10 Sign-in / sign-up page

It must have two forms: one for sign-in and one for sign-up.

4.11 Forms design

  • Teams will design the visual appearance of the web forms present in the website (sign-up, sign-in and contact forms).
  • You have to include a minimum of 15 input fields (buttons do not count) among the three forms.
  • You must use different types of fields (the more, the better): select, radio, checkbox, textarea,…
  • Additionally, you can use HTML5 input fields: date, email, number,…
  • Form fields must be aligned.
  • You must use:
    • label tags
    • fieldset tags
    • placeholder attribute
  • You should honor the semantic structure of HTML: e.g., you must not put labels and inputs in different layers.
  • Forms must be responsive:
    • Mobile devices
      • labels must be hidden.
      • radio and checkbox labels must be visible.
      • Input fields must occupy the entire screen width.
      • Forms must be centered.
    • Tablets
      • labels must be shown above fields.
      • radio and checkbox labels must be visible.
      • You can put more than one field in one line if you want.
    • Rest of screens
      • labels must be shown left side of the fields.
      • labels must be right aligned.
      • All labels must be visible.
Definition of Done
Forms meet the design requirements. Labels and controls are aligned. All team members are satisfied with the design. The design looks professional.

5 Hints

  • Remember to use the GIT workflow proposed for teams.
  • Do not create new CSS unless necessary. You should assign Bootstrap class names to elements instead.
  • Teams must create tasks that generate valid product updates (e.g., a task should generate a complete web page available in the web server).
  • Teams should agree on a base HTML template for all files. This task should be the first to work on. The structure can be changed during the Sprint, however.
  • It is advisable not to work more than one person in the same file to minimize conflicts. Tasks should affect different files.
  • You should do one commit and one push at least once in every class.

6 Grading criteria

Scrum board (Flip) (20%)
Daily update, tasks estimation updates, Burn Down chart, daily scrums.
Percentage of completed Product Backlog (20%)
Completed tasks.
Quality (20%)
Originality, design, validation tests, correct visualization in different browsers and devices, design consistency, usability and accessibility features.
Retrospective (20%)
Correct revision of processes, identification of strengths and weaknesses, team analysis, individual analysis.
Demo session (20%)
Participation of all team members, oral presentation, materials used in the presentation, adaptation to available time.

Individuals who do not attend the Eduscrum events (planning meeting, demo, retrospective) will get the corresponding decrease in their grade.
